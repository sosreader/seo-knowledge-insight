# Implementation Plan: 週報知識庫引用附上 Notion 連結

> Generated by /design on 2026-03-03
> Status: PENDING APPROVAL

## Overview

在 SEO 週報的知識庫引用部分加入 Notion 原始頁面連結，讓讀者一鍵回到 Notion 追溯原始會議紀錄。採用 **Enrichment 層注入** 方案，重用現有 `_enrichment` 機制，不修改 Step 2/3 pipeline。

## 方案選擇

| 維度 | 方案 A（全鏈路傳遞） | 方案 B（Lazy Lookup） | **方案 C（Enrichment 注入）** |
|------|---------------------|---------------------|---------------------------|
| 改動範圍 | 3 個 pipeline 腳本 | 1 個 report 腳本 | **1 個 enrich 腳本 + store** |
| 需重跑 pipeline | Step 2+3 | 不需要 | **只需 `make enrich`** |
| 資料自包含 | 是 | 否 | **是** |
| 與現有架構一致性 | 中等 | 低 | **高** |
| Supabase 遷移友好 | 是 | 需額外處理 | **是** |

**選擇方案 C**：重用 `_enrichment` 子物件（已有 synonyms / freshness_score），最小改動、零 LLM 成本。

## Architecture Notes

**為何不修改 Step 2/3**：
- Step 2 (`extract_qa`) 的職責是「從文本萃取 Q&A」，URL 解析違反 Single Responsibility
- Step 2 的 content-addressed cache 不含 source metadata（設計正確），不需清 cache
- `_enrichment` 機制已成熟（synonyms、freshness_score、search_hit_count），notion_url 自然歸位

**Supabase 遷移友好**：`qa_enriched.json` 是 QAStore 唯一入口，所有欄位一併進入資料庫。

**合併 Q&A 處理**：15 筆 `is_merged=True` 的 Q&A 取 `source_file` 對應的 URL（代表性來源）。

## Risks & Mitigations

| 風險 | 影響 | 對策 |
|------|------|------|
| `meetings_index.json` 的 `md_file` 與 `source_file` 格式不一致 | 映射失敗 | `Path(md_file).name` 正規化 + fallback 讀 Markdown 內容 |
| LLM 生成週報時移除連結格式 | 引用無法點擊 | `REPORT_SYSTEM_PROMPT` 明確指示 + best-effort |
| `meetings_index.json` 不存在 | enrichment 失敗 | 雙層 fallback：index → Markdown 內容 → 空字串 |
