## Implementation Steps

### Phase 1: Observability + Quick Wins（1-2 週）

> 目標：建立品質防線 + 成本可見度 + 回饋閉環

#### Step 1.1: CI Eval Gate — PR 品質門禁

**業界參照**: Braintrust 的 eval-driven development — 每次 PR 自動跑 eval，低於 threshold block merge。

- [ ] **1.1.1** 在 `evals/eval_retrieval.py` 結尾加入 threshold assertion
  - `assert hit_rate >= 0.95, f"Hit Rate {hit_rate} < 0.95"`
  - `assert mrr >= 0.90, f"MRR {mrr} < 0.90"`
  - CI 自然 fail（Python `sys.exit(1)`），不需額外基礎設施
  - File: `evals/eval_retrieval.py`

- [ ] **1.1.2** 在 `evals/eval_extraction.py` 加入 extraction quality threshold
  - File: `evals/eval_extraction.py`

- [ ] **1.1.3** 建立 `eval/eval_thresholds.json` 集中管理閾值
  - 格式: `{ "retrieval": { "hit_rate": 0.95, "mrr": 0.90 }, "extraction": { "accuracy": 0.80 } }`
  - File: `eval/eval_thresholds.json`（新建）

#### Step 1.2: Cost/Latency Monitoring — LLM 使用追蹤

**業界參照**: OpenTelemetry `gen_ai` semantic conventions — 每個 LLM call 記錄 token/cost/latency。

  - ECC: database-reviewer (sonnet)

- [ ] **1.2.2** 建立 `llm-usage-logger.ts` — fire-and-forget 寫入

- [ ] **1.2.3** 新增 `GET /api/v1/admin/usage?days=30` endpoint

#### Step 1.3: Exact Match Response Cache

**業界參照**: Content-addressed caching — `sha256(normalize(query))` 做 key，deterministic、測試容易、不需額外 embedding 呼叫。

**與 cache-redis.md 的關係**: 替代 Phase 1 的 disk cache，用 Supabase table 替代本地檔案。Redis（Phase 2）作為未來 scale out 路徑保留。

- [ ] **1.3.1** Supabase migration: `response_cache` table
  - Schema: `{ query_hash text PRIMARY KEY, query_text text, response jsonb, created_at timestamptz, expires_at timestamptz, hit_count int DEFAULT 0 }`
  - 不使用 vector — 純 text hash lookup，零額外 embedding 成本
  - File: `api/supabase/migrations/` 新 migration

- [ ] **1.3.2** 建立 `response-cache.ts` service
  - `normalizeQuery(q)`: lowercase + trim + 去除標點
  - `lookupCache(queryHash)`: 查 Supabase + 檢查 expires_at
  - `saveCache(queryText, queryHash, response, ttlHours=2)`
  - File: `api/src/services/response-cache.ts`（新建，~60 行）

- [ ] **1.3.3** 整合至 `rag-chat.ts`
  - 在 `ragChat()` 開頭：無 history 時做 cache lookup
  - 命中：直接回傳 + `metadata.cache_hit: true`
  - 結尾：save cache
  - File: `api/src/services/rag-chat.ts`

- [ ] **1.3.4** 測試
  - File: `api/tests/services/response-cache.test.ts`（新建，~80 行）
  - ECC: tdd-guide (sonnet)

### Phase 2: RAG 品質 + UX（2-3 週）

> 目標：提升回答精確度、可追溯性、感知速度

#### Step 2.1: Score-based Quality Gate（Rule-based CRAG）

**業界參照**: Corrective RAG (ICLR'26) — retrieval quality gate 偵測低品質 context。採用 rule-based 實作避免額外 LLM 延遲，適配 Lambda 120s timeout。

- [ ] **2.1.1** 在 `rag-chat.ts` 的 `formatContext` 前加入 3-tier threshold 判斷
  - `score >= 0.6` → **correct** → 正常 RAG
  - `0.4 <= score < 0.6` → **ambiguous** → RAG + disclaimer prompt（「以下結果可能不完全相關」）
  - `score < 0.4` → **incorrect** → `mode: "context-only"` fallback + `recordMiss()`
  - 新增 `metadata.retrieval_quality: "correct" | "ambiguous" | "incorrect"`
  - Laminar scoreEvent: `retrieval_gate_result`
  - File: `api/src/services/rag-chat.ts`

- [ ] **2.1.2** 測試：mock 3 種分數場景
  - File: `api/tests/services/rag-chat.test.ts`（擴充）

- [ ] **2.1.3** eval 新增 `retrieval_gate_accuracy` 指標
  - File: `evals/eval_chat.py`

#### Step 2.2: Inline Citation

**業界參照**: Perplexity/Gemini citation UX + RAGAS citation metrics — 每句話標注來源 Q&A ID。

- [ ] **2.2.1** 修改 RAG system prompt 要求 `[1][2]` 標注
  - File: `api/src/services/rag-chat.ts`（`SYSTEM_PROMPT`）
  - File: `api/src/agent/agent-loop.ts`（`SYSTEM_PROMPT`）

- [ ] **2.2.2** 後處理 `validateCitations(answer, sourceCount)`
  - Regex 擷取 `\[\d+\]`，驗證 N 在 1~sourceCount 範圍
  - `metadata.citation_count: number`
  - Laminar scoreEvent: `citation_count`

- [ ] **2.2.3** eval 新增 `citation_coverage` 指標（目標 >= 0.70）

### Phase 3: 報告智能進化（1-2 月）

#### Step 3.1: 時序異常偵測

**業界參照**: Semrush/Ahrefs anomaly detection — AI 驅動的持續監控，從「本週 vs 上週」升級為歷史脈絡感知。

- [ ] **3.1.1** 建立 `timeseries-analyzer.ts`
  - **Moving Average Deviation**: `value < MA(4w) * 0.85` → anomaly
  - **Consecutive Decline**: >= 3 consecutive weeks declining → alert
  - **Trend Detection**: linear regression slope 顯著為負 → declining trend
  - 輸入: `MetricsSnapshot[]`（多週快照序列）
  - 輸出: `AnomalyResult { metric, type, severity, context }`
  - File: `api/src/services/timeseries-analyzer.ts`（新建，~200 行）

- [ ] **3.1.2** 整合至報告生成

- [ ] **3.1.3** 新增 API endpoint

## Architecture Notes

### Architect 審查結果

1. **精簡方案優先**：原始 4 Phase 精簡為 3 Phase。GraphRAG（過度工程）、Topic Clustering（規模不足）、Semantic Cache pgvector（過度複雜）已移除
2. **Lambda 限制**：CRAG quality gate 採 rule-based（非 LLM），避免 120s timeout 衝突；clustering 類計算不在 Lambda 執行
3. **Supabase 容量**：新增 2 tables（`response_cache` + `llm_usage`），估計增加 < 10MB，免費方案仍安全
4. **安全性**：AI Crawler Readiness 降級為 CLI only（SSRF 風險）；cache 需考慮 cache poisoning（只 cache authenticated 請求）

| 項目 | 理由 | 重新評估條件 |
|------|------|-------------|
| GraphRAG | 1,323 筆 QA 規模太小，entity extraction 成本高 | QA > 5,000 筆 + multi-hop 問題佔比 > 20% |

## Risks & Mitigations

| Risk | 嚴重度 | Phase | Mitigation |
|------|--------|-------|------------|
| CI eval gate 增加 PR merge 時間 | Low | 1 | eval 只跑最關鍵的 2 steps (~30s) |
| Cache 回傳過時答案 | Medium | 1 | TTL 2 小時 + 知識庫更新時 invalidate |
| Cache poisoning（惡意 query 寫入） | Medium | 1 | 只 cache authenticated 請求 |
| CRAG threshold 過嚴 → 太多 fallback | Medium | 2 | 從寬鬆開始 (0.35)，根據 false positive rate 調校 |
| Inline citation LLM 不遵循 | Low | 2 | 後處理驗證 + scoreEvent 監控 |
| Lambda Streaming 與 Hono 整合問題 | Medium | 2 | 先做可行性研究，不行就延後 |
| 時序異常偵測假陽性 | Medium | 3 | >= 4 週資料才啟用 + 保守 threshold |
| AI visibility 資料不可得 | High | 3 | Graceful degradation |
| Supabase 免費方案容量 | Low | All | 新增 < 10MB，持續監控 |
