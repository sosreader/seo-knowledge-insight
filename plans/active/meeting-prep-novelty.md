# Implementation Plan: Meeting-Prep Content Novelty — SITREP + Changelog 混合模式

> Generated by /design on 2026-03-29
> Status: PENDING APPROVAL

## Context

連續週次的 meeting-prep 報告在 S2（業界動態）、S4（交叉比對）、S6/S8（評分）、S7（人本要素引用）出現高度重複，降低報告實際可用性。現有 eval 體系（5 個 eval 腳本、9+ 維度）**無任何跨報告 novelty/repetition 偵測**。

**業界參考**：軍事 SITREP（FM 101-5-2）以「自上次報告以來的變化」為核心，所有項目標注 NEW/ONGOING/RESOLVED；管理顧問（McKinsey/Bain）用 RAG + Carry-Forward 模式；軟體業用 Changelog（Added/Changed/Removed）。

本計畫採 **SITREP + Changelog 混合模式**，三軌並行：
1. **結構軌**：從報告結構本身消除重複（SITREP 標注 + Changelog 分類 + toggle 折疊區塊）
2. **Eval 軌**：新增 `eval_meeting_prep_novelty.py`（rule-based，零 LLM 成本），4 個新維度偵測跨週重複
3. **生成軌**：修改 `meeting-prep.md` skill，注入 delta awareness

## Implementation Steps

### Phase 1: 報告結構重設計（SITREP + Changelog + Toggle）

**Step 1: 修改 meeting-prep.md — Section 級 SITREP 標注規則**

在 Step E 的 section 生成指令中，為每個高重複風險 section 加入標注和呈現規則：

```
| Section | 模式 | 呈現方式 |
|---------|------|---------|
| S0 摘要 | SITREP | 只列「本週新增」的 5 bullets，不 carry forward 上週 |
| S2 業界 | SITREP | 每個動態項標注 [NEW] / [ONGOING-W{n}] / [RESOLVED] |
| S3 假設 | Changelog | 分為 New / Updated / Validated / Discarded 四類 |
| S4 比對 | RAG+CF | topic 標注 NEW 或 CF + 本週進展 |
| S6 EEAT | Changelog | 只展開 Changed 維度；No Change 一句帶過 |
| S7 人本 | Changelog | 同上 |
| S8 成熟度 | Changelog | 同上 |
| S9 提問 | SITREP | 標注 [NEW] 或 [CARRY-W{n}]（上週遺留第 n 週） |
```

**Step 2: 修改 meeting-prep.md — Toggle 折疊模板**

為 S6/S7/S8 中「No Change」的維度定義 toggle 折疊區塊（HTML `<details>` tag，GitHub/前端均可渲染）：

```markdown
#### S6 E-E-A-T 變更摘要

**Changed this week:**
| 維度 | 分數 | 變化 | 原因 |
|------|------|------|------|
| Authoritativeness | 2→3 | ↑ | 外部連結回穩，Yahoo 引用已確認恢復 |

<details>
<summary>No Change（3 維度，點擊展開上週評估）</summary>

| 維度 | 分數 | 上週依據（carry forward） |
|------|------|------------------------|
| Experience | 3 | UGC 平台作者署名仍不一致... |
| Expertise | 3 | 攻略類有深度但影評類缺獨到分析... |
| Trustworthiness | 3 | 定期更新但 AMP 警告持續攀升... |

</details>
```

同樣模式套用到 S7（7 要素）和 S8（成熟度）：Changed 維度展開、No Change 維度 toggle 收合。

S2 的 toggle 用於 ONGOING 項目：
```markdown
### 業界動態

**[NEW] Google March 2026 Spam Update 完成部署（3/24-25）**
- 影響 Ranking，可能強化 UGC 品質篩選...

**[NEW] 9to5Google: Google Search referrals plummeted**
- Google 搜尋導流大幅下滑，AI 來源仍不足 1%...

<details>
<summary>[ONGOING-W2] March 2026 Core Update — 部署中（本週新發展）</summary>

上週已報告：3/27 啟動，預計 2 週完成。
**本週進展**：早期數據顯示部分網站 organic traffic 下降 20-35%。Discover 組件影響開始浮現。

</details>
```

S4 交叉比對的 toggle 用於 Carry-Forward topic：
```markdown
| 狀態 | 主題 | KB | 顧問 | 指標 | 業界 | 判斷 |
|------|------|-----|------|------|------|------|
| **[NEW]** | March Spam Update vs UGC | ... | ... | ... | ... | 需驗證 |
| **[NEW]** | Video 爆發與影評轉型 | ... | ... | ... | ... | 缺口 |

<details>
<summary>[CF] 3 個延續主題（點擊展開本週進展）</summary>

| 主題 | 上週判斷 | 本週進展 |
|------|---------|---------|
| Discover 演算法 | 外部因素為主 | Discover 反彈 +56.5%，假設成立 |
| AI 流量趨勢 | 一致（正向關聯） | GPT +35% 持續驗證，無新矛盾 |
| 外部連結流失 | 需確認 Yahoo | 待 Ahrefs 驗證，暫無進展 |

</details>
```

**Step 3: 新增 Step C-prime — 前週報告差異偵測**

在 Step C（KB 搜尋）和 Step D（顧問文章）之間插入（~20 行）：

```markdown
### Step C-prime：前週報告差異偵測（自動執行）

Glob: output/meeting_prep_*.md → 取日期最新且早於本次的一份
Read 前週報告的 S2/S4/S6/S7/S8 段落，建立 prevReport 物件：

- prev_s2_items：每個動態項的 {url, title, status}（用於 SITREP 標注）
- prev_s4_topics：S4 表格的主題欄 + 判斷欄（用於 CF 標注）
- prev_s6_scores：E-E-A-T 4 維度分數 + 依據摘要
- prev_s7_scores：人本 7 要素分數 + 引用 citation IDs
- prev_s8_levels：成熟度 4 維度等級 + 依據摘要
- prev_s9_questions：S9 提問 ID 列表（用於 CARRY 標注）

若找不到前一份，所有項目標注 [NEW]，不使用 toggle 折疊。

**SITREP 標注規則：**
- S2 動態項：URL 完全相同 → [ONGOING-W{n}]（n = 連續出現週數）；URL 不同 → [NEW]；上週有但本週移除 → [RESOLVED]
- S4 topic：主題名稱相同 → [CF]；新增 → [NEW]
- S6/S7/S8 維度：分數相同 → No Change（收入 toggle）；分數不同 → Changed（展開）
- S9 提問：問題語意相似（包含相同指標名稱 + 相同 section 來源）→ [CARRY-W{n}]；否則 → [NEW]
```

**Step 4: 修改 Step E 各段的 delta 指引**

| Section | 新增指引 | 說明 |
|---------|---------|------|
| S2 | ONGOING 項用 toggle 折疊，只寫「本週新發展」；NEW 項完整展開 | 消除 Google 更新跨週重複 |
| S3 | 假設分 4 類：New Hypothesis / Updated（修正假設）/ Validated（會議確認）/ Discarded（數據否定） | 追蹤假設生命週期 |
| S4 | NEW topic ≥ 2 個；CF topic 用 toggle，只寫「本週進展」 | 消除相同 KB 引用重複 |
| S6 | Changed 維度展開 + ↑↓ 標注；No Change 維度 toggle 折疊 + carry forward 上週依據 | 分數相同時不浪費版面 |
| S7 | 同 S6 | 同上 |
| S8 | 同 S6 | 同上 |
| S9 | NEW 提問完整展開；CARRY 提問 toggle 折疊 + 補充「本週新 context」 | 避免連續問相同問題 |

### Phase 2: Eval Script + Tests（TDD）

**Step 5: 先寫測試** `tests/test_eval_meeting_prep_novelty.py`

12 個 test cases：
- `test_find_prior_report_*`（2）：正常找到 / 第一份回傳 None
- `test_s2_novelty_*`（3）：完全不同=1.0 / 完全相同=0.0 / 部分重疊
- `test_s4_novelty_*`（2）：topic 提取正確 / CF 項不計入 novelty
- `test_score_drift_*`（2）：完全靜止=0.0 / 有變化>0
- `test_s7_citation_novelty_*`（2）：全新=1.0 / 全重複=0.0
- `test_toggle_detection`（1）：`<details>` 內的內容不參與 novelty 計算

**Step 6: 實作 eval 腳本** `evals/eval_meeting_prep_novelty.py`

4 個維度 + 1 個結構分：

| 維度 | 演算法 | 閾值 | 說明 |
|------|--------|------|------|
| `s2_novelty` | S2 中 `[NEW]` 項數 / S2 總項數 | ≥ 0.40 | 至少 40% 為新動態 |
| `s4_novelty` | S4 中 `[NEW]` topic 數 / S4 總 topic 數 | ≥ 0.30 | 至少 30% 為新 topic |
| `score_drift_novelty` | E-E-A-T + Maturity 分數變化量 / 4.0 | ≥ 0.10 | 懲罰完全複製貼上 |
| `s7_citation_novelty` | S7 引用中非 toggle 內的新 stable_id 比例 | ≥ 0.30 | KB 引用新鮮度 |
| `toggle_structure` | 報告中是否正確使用 `<details>` 折疊 No Change 內容 | ≥ 0.50 | 結構合規性 |

**演算法修正**：有了 SITREP 標注後，s2_novelty 和 s4_novelty 可以直接計算 `[NEW]` 標籤比例（比 URL Jaccard 更精準）。Fallback：若報告未使用標注格式（舊報告），退回 URL Jaccard。

核心函式：
- `_find_prior_report(path) -> Path | None`：同目錄找最近前一份
- `_count_sitrep_tags(section: str) -> dict[str, int]`：計算 NEW/ONGOING/CF/RESOLVED/CARRY 各標籤數量
- `_extract_toggle_content(section: str) -> tuple[str, str]`：分離 toggle 內（No Change）和 toggle 外（Changed）的內容
- 5 個 scorer + `executor()` + Laminar `evaluate()` 整合
- no_prior（第一份報告）：所有維度 1.0

**Step 7: 更新 golden dataset** `eval/golden_meeting_prep.json`
- 新增 `"expected_novelty"` 欄位（保守閾值）
- 注意：現有 4 個 fixtures 使用舊格式（無 SITREP 標注），`toggle_structure` 應設 0.0 或跳過

**Step 8: 更新 Makefile**
- 新增 `evaluate-meeting-prep-novelty` target
- 加入 `evaluate-meeting-prep-full` 依賴鏈

## Toggle 折疊的前端相容性

| 環境 | `<details>` 支援 | 備註 |
|------|-----------------|------|
| GitHub Markdown | 完整支援 | PR/Issue/README 均可 |
| VS Code Preview | 完整支援 | 內建 markdown 渲染 |
| 前端 vocus-web-ui | 需確認 | 若使用 react-markdown，需 `rehype-raw` plugin |
| Supabase 儲存 | 無影響 | 純文字儲存，渲染由前端負責 |
| CLI `cat` / `less` | 原始 HTML 顯示 | 可讀但不折疊，不影響使用 |

若前端不支援 `<details>`，fallback 為直接顯示完整內容（不折疊），不會遺失資訊。

## Risks & Mitigations

| 風險 | 機率 | 對策 |
|------|------|------|
| `score_drift_novelty` 永遠 ~0（E-E-A-T 穩定） | 高 | 閾值 0.10 極低，只抓完全複製；觀察後可移除 |
| 同一 Core Update 跨 2 週，S2 URL 重疊 | 中 | SITREP [ONGOING] + toggle 折疊，展開只寫新發展 |
| 舊報告無 SITREP 標注，eval 退回 Jaccard | 中 | eval 腳本 dual-path：有標注用標注計數，無標注用 Jaccard |
| `<details>` 在前端不渲染 | 低 | Fallback 為直接顯示，不遺失資訊 |
| Claude Code 不遵守 toggle 格式指令 | 中 | eval `toggle_structure` 維度檢查合規，低分時 prompt 迭代 |
