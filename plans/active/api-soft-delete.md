## Overview

所有 DELETE 操作改為軟刪除（標記 `deleted_at` 而非物理刪除）。影響 4 張表（reports/sessions/metrics_snapshots/meeting_prep）、4 個 Supabase store class、3 個 DELETE 端點。synonym_custom 排除（設定資料，非使用者內容）。

## Architecture Notes (設計決策)

### 方案選擇

| 方案 | 優點 | 缺點 | 決策 |
|------|------|------|------|
| A: `deleted_at` column | 簡單、業界標準、可還原 | 所有 SELECT 需加 filter | **採用** |
| B: `is_deleted` boolean | 更簡單 | 無法追蹤刪除時間、無法排序 | 不採用 |
| C: Archive table | 乾淨分離 | 需維護兩套 schema、JOIN 複雜 | 不採用 |

### 關鍵設計

1. **RLS 層面過濾**：在 RLS policy 加 `AND deleted_at IS NULL`，讓 anon key 自動看不到已刪除資料——即使忘了在 query 加 filter 也不會洩漏
2. **Partial index**：`CREATE INDEX ... WHERE deleted_at IS NULL`，讓正常查詢效能不受影響
3. **Store 層改動最小化**：`delete()` 方法改為 `PATCH { deleted_at: now() }`，不改方法簽名
4. **可選 purge**：預留 `purge()` 方法（物理刪除），但不暴露端點——僅供未來管理員用
5. **synonym_custom 排除**：設定資料刪了就是刪了，無需追溯

## Implementation Steps

### Phase 1: Migration

- [ ] **Step 1**: 新增 `supabase/migrations/013_soft_delete.sql`
  ```sql
  -- Add deleted_at to 4 tables
  ALTER TABLE reports ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ NULL;
  ALTER TABLE sessions ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ NULL;
  ALTER TABLE metrics_snapshots ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ NULL;
  ALTER TABLE meeting_prep ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ NULL;

  -- Partial indexes for active rows (normal queries hit these, not full table)
  CREATE INDEX IF NOT EXISTS reports_active_idx
    ON reports (date_key DESC) WHERE deleted_at IS NULL;
  CREATE INDEX IF NOT EXISTS sessions_active_idx
    ON sessions (updated_at DESC) WHERE deleted_at IS NULL;
  CREATE INDEX IF NOT EXISTS snapshots_active_idx
    ON metrics_snapshots (created_at DESC) WHERE deleted_at IS NULL;
  CREATE INDEX IF NOT EXISTS meeting_prep_active_idx
    ON meeting_prep (date_key DESC) WHERE deleted_at IS NULL;

  -- Update RLS: anon SELECT only sees non-deleted rows
  -- reports
  DROP POLICY IF EXISTS "reports_read_all" ON reports;
  CREATE POLICY "reports_read_active" ON reports
    FOR SELECT USING (deleted_at IS NULL);

  -- sessions
  DROP POLICY IF EXISTS "sessions_read_all" ON sessions;
  CREATE POLICY "sessions_read_active" ON sessions
    FOR SELECT USING (deleted_at IS NULL);

  -- metrics_snapshots
  DROP POLICY IF EXISTS "snapshots_read_all" ON metrics_snapshots;
  CREATE POLICY "snapshots_read_active" ON metrics_snapshots
    FOR SELECT USING (deleted_at IS NULL);

  -- meeting_prep
  DROP POLICY IF EXISTS "meeting_prep_read_all" ON meeting_prep;
  CREATE POLICY "meeting_prep_read_active" ON meeting_prep
    FOR SELECT USING (deleted_at IS NULL);
  ```
  - Agent: `database-reviewer`（審查 index 策略和 RLS 正確性）

### Phase 2: Store 改動

- [ ] **Step 2**: 新增 `supabasePatch` helper 到 `supabase-client.ts`
  ```typescript
  export async function supabasePatch(
    table: string,
    queryString: string,
    data: Record<string, unknown>,
  ): Promise<void> {
    const url = `${config.SUPABASE_URL}/rest/v1/${table}${queryString}`;
    const resp = await fetch(url, {
      method: "PATCH",
      headers: supabaseHeaders(config.SUPABASE_SERVICE_KEY || config.SUPABASE_ANON_KEY),
      body: JSON.stringify(data),
      signal: AbortSignal.timeout(SUPABASE_TIMEOUT_MS),
    });
    if (!resp.ok) {
      const body = await readErrorBody(resp);
      throw new Error(`Supabase PATCH ${table} failed (${resp.status})${body}`);
    }
  }
  ```
  - 注意：使用 `readErrorBody()` 截斷（修復 security review 發現的 raw error body 洩漏）

- [ ] **Step 3**: 修改 4 個 store class 的 `delete()` 方法
  - Agent: `tdd-guide`（先更新測試 expectations，再改實作）
  - **SupabaseReportStore.delete()**：
    ```typescript
    async delete(dateKey: string): Promise<boolean> {
      const exists = await this.exists(dateKey);
      if (!exists) return false;
      await supabasePatch("reports", `?date_key=eq.${encodeURIComponent(dateKey)}`, { deleted_at: new Date().toISOString() });
      return true;
    }
    ```
  - **SupabaseSnapshotStore.delete()**：同上模式
  - **SupabaseSessionStore.deleteSession()**：改用 `supabasePatch` 取代 raw fetch DELETE
  - **SupabaseMeetingPrepStore.delete()**：同上模式 + `assertSafeDateKey()` 保留

- [ ] **Step 4**: 確認 `list()` 和 `getBy*()` 查詢
  - 因 RLS policy 已加 `deleted_at IS NULL`，anon key 的 SELECT 自動過濾
  - service_role 的查詢需手動加 `&deleted_at=is.null`（或不加——service_role bypass RLS）
  - **決策**：list/get 方法不改（RLS 處理），service_role 路徑若需要看全部再加 `?include_deleted=true` 參數

## Risks & Mitigations

| 風險 | 影響 | 對策 |
|------|------|------|
| RLS policy 名稱衝突 | DROP 舊 policy 失敗 | 用 `IF EXISTS` |
| service_role 看到已刪除資料 | 管理操作混淆 | service_role bypass RLS 是設計如此；若需過濾，query 手動加 filter |
| 舊 `supabaseDelete` 仍被呼叫 | 物理刪除逃逸 | Step 3 改完後 grep 確認無殘留 |
| Partial index 增加寫入 overhead | 微 | 資料量小（< 100 rows per table），不構成問題 |
| synonym_custom 不一致 | 使用者疑惑「為什麼同義詞刪了就沒了」 | 文件說明 synonym 為設定資料，與內容型資料策略不同 |
