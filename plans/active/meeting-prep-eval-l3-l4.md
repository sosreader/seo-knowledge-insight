## Architecture Notes (Architect 審查結果)

### 已採納的修改

1. **合併 6 次 API call 為 1 次**：一次傳送必要 sections + 6 維度 rubric，要求 LLM 輸出 `{"dimensions": {...}}` JSON。成本從 $0.007 降至 $0.002/次，且跨維度一致性更好
2. **`overall_coherence` 必須包含 S0+S1+S3+S9** 四個 section 的完整內容
3. **`url_accessibility_rate` 分母為零時回傳 None**，CI gate 自動跳過
4. **Makefile 語意分離**：`evaluate-meeting-prep` 保持 L1+L2（零成本），新增 `evaluate-meeting-prep-full` = L1+L2+L3+L4
5. **L3 跳過時輸出可見 log**：`[eval_meeting_prep_llm] SKIPPED: ANTHROPIC_API_KEY not set`
6. **`source_diversity` 加規範化 map**：`{"SER": "seroundtable", "SEL": "searchengineland", ...}`
7. **與 `/evaluate-meeting-prep-quality` slash command 互補說明**：腳本 = CI batch gate，slash command = 互動式 + 改善建議

### 延後至 v2 的項目

- 加入 2 個「刻意有缺陷」的反例 fixture（s3 全泛用、s4 全一致）
- L3 prompt 加 few-shot calibration examples
- Slash command 與 L3 腳本的 rubric 同步機制

### 文件影響

- `CLAUDE.md`：新增 Makefile targets 說明
- `eval/eval_thresholds.json`：新增 L3/L4 閾值
- `.env.example`：ANTHROPIC_API_KEY 備註更新
- `.claude/commands/evaluate-meeting-prep-quality.md`：加 CI 版本互參
- `memory/MEMORY.md`：eval group 數 13→15

## Implementation Steps

### Phase 1: Golden Dataset 擴充

- [ ] **Step 1**: 修改 `eval/golden_meeting_prep.json`
  - 為每個 case 加入 `expected_llm` 欄位：
    ```json
    "expected_llm": {
      "min_avg_score": 3.0,
      "min_score_per_dimension": {
        "s3_hypothesis_grounded": 3.0,
        "s6_eeat_justified": 3.0,
        "s9_question_specificity": 3.0,
        "s4_contradiction_quality": 2.5,
        "overall_coherence": 3.0,
        "s8_maturity_justified": 3.0
      }
    }
    ```
  - 為每個 case 加入 `expected_web` 欄位：
    ```json
    "expected_web": {
      "date_window_days": 30,
      "min_source_diversity": 2,
      "url_check_enabled": false
    }
    ```
  - L1/L2 腳本只讀 `expected_structure`/`expected_grounding`，不受影響

### Phase 2: L3 LLM-as-Judge 腳本

- [ ] **Step 2**: 建立 `evals/eval_meeting_prep_llm.py`（~300 行）
  - Agent: `docs-lookup`（查 Anthropic Messages API 結構化輸出）
  - 頂層 guard：`ANTHROPIC_API_KEY` 未設定 → stderr warning + `sys.exit(0)`
  - **單次 API call 設計**（architect 建議）：
    - 提取報告中 S0+S1+S3+S4+S5+S6+S7+S8+S9 sections
    - 一次送 system prompt（含 6 維度 rubric + 評分標準）+ user message（報告 sections）
    - 要求輸出 `{"dimensions": {"s3_hypothesis_grounded": {"score": N, "reason": "..."}, ...}}`
    - Model: env `JUDGE_MODEL`，fallback `claude-haiku-4-5-20251001`
    - `temperature=0`
  - Executor: 讀 .md → 提取 sections → 呼叫 Anthropic → 解析 JSON
  - 7 個 evaluator functions：6 維度各一（`score / 5.0`）+ `avg_score_above_threshold`
  - CI threshold gate（讀 `eval_thresholds.json["meeting_prep_llm"]`）
  - Laminar group: `meeting_prep_llm`
  - CLI: `--report <path>`、`--limit N`

### Phase 3: L4 Web Research Freshness 腳本

- [ ] **Step 3**: 建立 `evals/eval_meeting_prep_web.py`（~250 行）
  - **S2 section 解析器**：
    - `_extract_s2_section(content: str) -> str`：提取 S2 業界動態全文
    - `_parse_dates(s2_text: str) -> list[date]`：多格式支援（ISO / 斜線 / 中文）
    - `_parse_sources(s2_text: str) -> list[str]`：規範化 map 去重
    - `_parse_urls(s2_text: str) -> list[str]`：`[text](url)` pattern
  - `_extract_report_date(content: str) -> date`：從 H1 或 meta JSON 取得
  - 4 個 evaluator functions：
    - `date_freshness_rate`：dates within ±N days / total dates（無日期 → 0.5 neutral）
    - `source_diversity`：unique sources / min_source_diversity（cap 1.0）
    - `url_accessibility_rate`：enabled=false → 1.0 skip；分母=0 → None skip；否則 HEAD 200 ratio
    - `s2_content_density`：有效行數 >= 5 → 1.0；否則 lines/5
  - CI threshold gate
  - Laminar group: `meeting_prep_web`
  - CLI: `--report <path>`、`--limit N`

## Risks & Mitigations

| 風險 | 影響 | 對策 |
|------|------|------|
| LLM 評分隨機性 | L3 分數不穩定 | temperature=0 + 保守閾值（0.4-0.55） |
| S2 格式不統一 | L4 解析失敗 | 多 regex fallback + 無法解析回 0.5 neutral |
| Anthropic SDK 網路超時 | L3 失敗 | try/except + score=1 fallback |
| Fixture URL 已失效 | L4 url check 失敗 | url_check_enabled=false（初始） |
| L3 與 slash command 維度分裂 | 評分標準不一致 | 在兩處加互參說明 |
