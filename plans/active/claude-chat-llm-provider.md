## Overview

新增 `CHAT_LLM_PROVIDER=anthropic` 環境變數，讓 chat 功能（RAG / streaming / agent mode）透過 Anthropic Claude 生成回答，取代 OpenAI GPT-5.2。Embedding 仍使用 OpenAI（Anthropic 無 embedding API）；若無 OpenAI key 則退回 keyword-only search + Claude 生成。現有 OpenAI 使用者零影響。

### Phase 3: 新 Service 實作（TDD — 先寫測試）

  - JSON Schema 內容與 `getOpenAITools()` 完全相同
  - ~30 行

- [ ] **Step 10: 建立 `anthropic-chat.ts`** (RAG non-streaming)
  - File: `api/src/services/anthropic-chat.ts` (~150 行)
  - 與 `ragChat` 同 signature、同 retrieval pipeline（embedding → hybridSearch → rerank → quality gate）
  - LLM call 改用 Anthropic SDK（dynamic import，沿用 reranker.ts 的 pattern）
  - **關鍵差異**：
    - `system` 為 top-level 參數，不是 message（不能直接用 `buildMessages()` 輸出）
    - 策略：直接呼叫 `buildSystemPrompt()` 取得 system string，`buildMessages()` 的輸出過濾掉 `role: "system"`，只保留 user/assistant
    - Response: `resp.content[0].text`（不是 `choices[0].message.content`）
    - Usage: `resp.usage.input_tokens` / `resp.usage.output_tokens`
  - `buildMetadata({..., provider: "anthropic"})`
  - **若無 OpenAI key**：跳過 embedding，用 `qaStore.keywordSearch()` 替代 `hybridSearch()`
  - Export `anthropicRagChatObserved = observe("anthropic_rag_chat", ...)`

- [ ] **Step 11: 建立 `anthropic-chat-stream.ts`** (streaming)
  - File: `api/src/services/anthropic-chat-stream.ts` (~150 行)
  - 沿用 `StreamCallbacks` interface
  - Anthropic streaming: `client.messages.stream({...})`
  - Token event: `event.type === "content_block_delta"` → `event.delta.text`
  - Final usage: `await stream.finalMessage()` → `usage.input_tokens/output_tokens`
  - metadata `provider: "anthropic"`

- [ ] **Step 12: 建立 `anthropic-agent-loop.ts`** (agent mode)
  - File: `api/src/agent/anthropic-agent-loop.ts` (~220 行)
  - 與 `agentChat` 同 signature + `AgentResponse` 回傳型別
  - 使用 `getAnthropicTools()` 而非 `getOpenAITools()`
  - **關鍵差異**：
    - 終止: `stop_reason === "end_turn"` 且無 `tool_use` content block
    - Tool use: `resp.content.filter(b => b.type === "tool_use")` → 取 `block.id`, `block.input`
    - Tool result: `{ role: "user", content: [{ type: "tool_result", tool_use_id, content }] }`
    - History 必須 user/assistant 交替 — 需 normalize
    - `system` 為 top-level 參數
  - metadata `provider: "anthropic"`
  - Risk: **High** — tool_use protocol 與 OpenAI function calling 差異大

## Architecture Notes

### 設計決策

1. **不建抽象層** — 不做 `LLMClient` interface。改為建立平行的 Anthropic service files（`anthropic-chat.ts` / `anthropic-chat-stream.ts` / `anthropic-agent-loop.ts`），在 route handler 層以 `getChatProvider()` 做 dispatch。理由：
   - 兩個 SDK 的 API 差異大（message format、streaming protocol、tool use），強制統一反而增加複雜度
   - 平行實作更容易獨立測試和除錯
   - 符合專案「小檔案、高內聚」原則

2. **Embedding 保留 OpenAI** — Anthropic 沒有 embedding API。若無 OpenAI key，退回 keyword-only search。這是暫時限制，未來可引入開源 embedding（如 Voyage AI）。

3. **Dynamic import** — 沿用 `reranker.ts` 的 `await import("@anthropic-ai/sdk")` 模式，避免無 Anthropic key 時載入 SDK 的開銷。

### 訊息格式轉換

```
OpenAI format (buildMessages 輸出):
  [{ role: "system", content: systemPrompt },
   { role: "system", content: "--- 知識庫 ---\n..." },
   ...history,
   { role: "user", content: message }]

Anthropic format (anthropic-chat.ts 內轉換):
  system: systemPrompt + "\n\n--- 相關 SEO 知識庫 ---\n..." + "\n--- 知識庫結束 ---"
  messages: [...history(user/assistant only), { role: "user", content: message }]
```

## Risks & Mitigations

| 風險 | 嚴重度 | 緩解措施 |
|------|--------|---------|
| `sessions.ts:166` `caps.llm === "openai"` 漏掉 Anthropic → silent context-only | **High** | Step 14 修正 + 專屬測試 |
| Anthropic tool_use protocol 與 OpenAI function calling 差異大 | **High** | TDD + 參考官方文件 + 多邊界測試 |
| `buildMessages` 回傳 `role: "system"` 被 Anthropic reject | **Medium** | anthropic-chat.ts 直接組合 system string，不用 buildMessages |
| History 非交替 role 被 Anthropic reject | **Medium** | normalize 函式 + edge case 測試 |
| `isAgentEnabled()` 改用 `hasChatLLM()` 後 auto-mode 行為變更 | **Low** | 只在有 Anthropic key 時才啟用，測試覆蓋 |
