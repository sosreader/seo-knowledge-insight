## Overview

本計畫以目前 main repo 狀態為準，目標是把模型設定、外部資料來源接入、API/CLI 行為、文件描述與 provenance metadata 對齊，但不做無差別的全域統一。依目前程式碼狀態，執行期預設模型已經大致切到 `gpt-5.4-nano`、`gpt-5.4` 與 `text-embedding-3-small`；本次重點是建立明確 use-case matrix、修補資料鏈路缺口、清理未使用設定，並避免在 dedupe 或 backfill 時把歷史 metadata 壓平成單一最新模型字串。

## Community Consensus & Dissenting Views

| 技術決策 | 社群主流共識 | 反面意見或已知陷阱 | 本計畫採納方式 |
| --- | --- | --- | --- |
| 依 use case 分流模型 | LLM app 維運普遍支持 extraction、report、chat 分開配置，因品質與成本需求不同 | env var 數量增加，文件若不同步會更難維護 | 只保留有實際 runtime consumer 的分流，並用 matrix 與測試鎖住 |

## Architecture Notes

| Area | Decision | Notes |
| --- | --- | --- |
| 保留分流的 config | `OPENAI_MODEL` 保留給 extraction 與 merge；`CLASSIFY_MODEL` 保留給分類；`REPORT_MODEL` 保留給 Python 與 TypeScript 報告生成；`CHAT_MODEL` 保留給 chat / agent；Anthropic reranker 保持獨立於 OpenAI 模型設定 | 這些分流都已有對應 executable consumer，不應因文件整理而被合併 |
| 可合併的 config 層級 | `OPENAI_EMBEDDING_MODEL` 維持單一共享設定；`OPENAI_MODEL`、`REPORT_MODEL`、`OPENAI_EMBEDDING_MODEL` 的 defaults 與說明可在 Python / TypeScript 文件層對齊 | 可合併的是 runtime contract 與文件表述，不是把 use case 合併成單一模型 |
| 不建議合併的範圍 | 不把 `REPORT_MODEL` 或 `CHAT_MODEL` 併回 `OPENAI_MODEL`；不把 Anthropic reranker 綁進 OpenAI env var；不做跨語言 mega-manifest 生成器 | 本次優先修 drift，不引入新的跨語言 build 耦合 |
| Python 來源清單權威點 | 以 `config.py` 集中 extract / state helper 所需的來源目錄清單，再由 `scripts/02_extract_qa.py` 與 `scripts/list_pipeline_state.py` 共同引用 | 這是修正 google blog / web.dev / screaming frog 漏接的根因 |
| TypeScript fetch 權威點 | 以 `api/src/routes/pipeline.ts` 內的單一 job descriptor 陣列描述 `fetch-articles` 的 9 個來源，回應結果與失敗收集都由同一陣列生成 | 避免再出現執行 3 個來源、文件寫 9 個來源的漂移 |
| Provenance 模型 | 保留 `extraction_model` 作為向下相容欄位；對 merged / backfilled QA 追加 additive provenance，而非覆寫原值 | `claude-code`、`claude-code-heuristic`、`local-heuristic` 必須原樣保留，可再加入 `derived_from_models` 或 `provenance_status` |
| 未使用設定處理 | `EVAL_RERANK_MODEL` 預設移除 | 目前沒有 executable consumer，保留只會誤導使用者與文件 |
| zh-TW 範圍界線 | 目前 main repo 僅含英文 Google Search Central Blog；`plans/active/google-blog-zhtw-source.md` 視為後續 phase 或外部依賴 | 本次計畫不得把 zh-TW 當成已接入來源，也不得在來源數量中算入第 10 個來源 |
| Pipeline 邊界 | 若不擴大 `scripts/run_pipeline.py` 行為，則文件必須明確區分 Notion-core pipeline 與 full-data pipeline | 推薦以 `Makefile` 的 `fetch-all` / `fetch-articles` 作為全來源入口，避免隱性改變現有腳本成本與時序 |

## Risks & Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| 目前 merged QA 會被寫成當前 `OPENAI_MODEL`，修正 provenance 時若設計不當可能破壞現有 API client | High | 讓 `extraction_model` 保持向下相容，新增 additive provenance 欄位；同步更新 file store 與 Supabase store loader |
| blanket backfill 改成 evidence-based 後，可能暴露一批無法可靠判定來源的歷史資料 | High | 將無法判定者明確標為 `legacy-unknown` 或等價 `provenance_status`，而不是偽造最新模型；同時輸出 dry-run bucket 統計 |
| API `fetch-articles` 從 3 來源擴到 9 來源後，執行時間與 partial failure 機率都會上升 | Medium | 保留 per-source result 與 partial failure 回應，不改成 all-or-nothing；測試穩定順序與失敗回報 |
| 無腦清理舊模型字串會破壞歷史 fixture 或 filter 行為測試 | Medium | 只更新 current default / public example 類測試；對歷史資料 fixture 改名或加註解說明用途 |
| 直接擴大 `scripts/run_pipeline.py` 可能改變既有工作流成本與期望 | Medium | 先在 Phase 1 做語義決策；若未經批准，不擴大腳本行為，只調整文件與入口說明 |
| 移除 `EVAL_RERANK_MODEL` 可能踩到未被審計到的外部腳本 | Low | 先做 repo-wide grep 與 targeted tests；若真的發現 consumer，改成同一 PR 補 executable code、文件與測試後再保留 |
