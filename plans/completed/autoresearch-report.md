## Overview

借鑑 autoresearch pattern，自動化優化 `/generate-report` 週報品質。與 retrieval 優化的關鍵差異：report 的「可修改物」是 prompt template（generate-report.md），每輪需要 LLM 生成報告再評分，因此每輪耗時 ~3-5 分鐘（vs retrieval 的 ~40 秒）。8 小時可跑 ~100-150 輪。

## 與 Retrieval 版本的關鍵差異

| 面向 | Retrieval | Report |
|------|-----------|--------|
| 可修改物 | TS 數值參數（25 個） | Prompt template（generate-report.md） |
| 每輪執行 | `runner.sh` → API search × 35 queries | Claude Code 生成報告 → evaluator 評分 |
| 每輪耗時 | ~40 秒 | ~3-5 分鐘 |
| 一晚實驗量 | ~400 輪 | ~100-150 輪 |
| Evaluator | 8 個 retrieval metrics | 5 個 report metrics（rule-based） |
| 資料需求 | golden_retrieval.json（固定） | metrics snapshot（需要真實指標資料） |

## Composite Score Formula

```
composite = (
  section_coverage       × 0.25 +   # 7 個 section 是否齊全
  kb_citation_count      × 0.20 +   # KB 引用數量（/6, capped 1.0）
  has_research_citations × 0.15 +   # 是否引用業界研究
  has_kb_links           × 0.15 +   # 是否有 /admin/seoInsight/{id} 連結
  alert_coverage         × 0.25     # ALERT 指標是否被行動清單提及
)
```

注意：`llm_augmented` 和 `has_crawled_not_indexed_section` 為獨立指標，不計入 composite。

## Agent 可修改的範圍

### `.claude/commands/generate-report.md`（唯一可修改檔案）

可調整的面向：
1. **Section 結構**：7 個 section 的標題、順序、子項目
2. **分析框架指引**：Health Score 演算法、異常閾值定義
3. **推理提示**：situation/traffic/technical/intent/action 的推理指引
4. **Citation 規則**：引用格式、數量要求
5. **Alert 覆蓋規則**：行動清單如何覆蓋 ALERT_DOWN 指標
6. **研究引用列表**：RESEARCH_KEYWORDS 對應的引用建議
7. **qaMap 搜尋策略**：廣域主題詞、top-k 數量

## Implementation Steps

### Phase 1: 建立 eval 基礎設施

- [ ] **Step 1**: 建立 `autoresearch/eval_report_local.py`（~120 行）
  - 讀取指定 report markdown 檔案
  - 從 report 內容解析 alert names（或從 CLI arg 傳入）
  - 呼叫 report-evaluator 的 Python port 邏輯（`_eval_report.py` 已有）
  - 計算 composite score
  - stdout 輸出 `REPORT_COMPOSITE=X.XXXX`

- [ ] **Step 2**: 建立 `autoresearch/runner_report.sh`（~100 行）
  - 流程：
    1. 選一個 snapshot（隨機或輪流使用 4 個可用 snapshot）
    2. 呼叫 Claude Code 生成報告：`claude -p "讀 autoresearch/program_report.md 的 generate 指令，用 snapshot <path> 生成報告到 /tmp/autoresearch_report.md"`
    3. 呼叫 `eval_report_local.py` 評分
    4. 解析 REPORT_COMPOSITE
    5. 寫入 results_report.tsv
  - **問題**：每輪需要 Claude Code 生成報告（~3 min），runner.sh 內無法呼叫 Claude Code
  - **解法**：runner 只負責 eval + TSV 記錄，報告生成由 agent 主循環直接執行

- [ ] **Step 3**: 建立 `autoresearch/results_report.tsv`（header）
  ```
  timestamp\tcomposite\tsection_coverage\tkb_citation\thas_research\thas_kb_links\talert_coverage\tgit_hash\tstatus\tdescription
  ```

### Phase 2: Agent 研究指令

- [ ] **Step 4**: 建立 `autoresearch/program_report.md`（~150 行）

  每輪流程（不同於 retrieval 的 runner.sh 模式）：

  ```
  LOOP FOREVER:
  1. 閱讀 autoresearch/results_report.tsv，了解已嘗試的修改和 best score
  2. 閱讀 .claude/commands/generate-report.md 目前內容
  3. 選擇一個改進假設（每次只改一個面向）
  4. 修改 generate-report.md
  5. git commit -m "experiment: <假設說明>"
  6. 隨機選一個 snapshot，用修改後的 prompt 生成報告：
     - 讀取 snapshot
     - 按 generate-report.md 流程生成報告到 /tmp/autoresearch_report_<hash>.md
  7. 用 _eval_report.py 評分：
     bash: .venv/bin/python scripts/_eval_report.py --report /tmp/autoresearch_report_<hash>.md --alert-names "<alerts>"
  8. 解析評分結果，寫入 results_report.tsv
  9. 判斷：
     - composite 提升 → 保留 commit
     - composite 退化 → git reset HEAD~1 --hard
  10. 回到步驟 1
  ```

  可修改面向 + 範圍 + 診斷提示（同 retrieval 的 program.md 風格）

## 架構問題：Report 自動循環的瓶頸

Report autoresearch 有一個 retrieval 版不存在的瓶頸：

**每輪需要 LLM 生成報告**。Retrieval 改參數後 tsx hot-reload 就能用 runner.sh 自動測，但 report 需要 Claude Code 自己讀 prompt → 讀 snapshot → 搜尋 KB → 推理生成 7 section → 存檔 → 跑 eval。

這代表 **agent 必須在循環中同時扮演「修改者」和「執行者」**——它改了 prompt，然後自己用改後的 prompt 生成報告。這在 autoresearch 原始設計中等同於 agent 自己改 train.py 然後自己跑 train.py。

### 建議方案

**方案 A（推薦）：Agent 內循環**
- Agent 在單一 session 中執行完整循環
- 不需要 runner.sh（agent 自己跑 eval Python script）
- 每輪：改 prompt → 自己生成報告 → 跑 eval → keep/discard
- 缺點：每輪 3-5 分鐘，context window 壓力大

**方案 B：雙 Agent**
- Agent A：改 prompt + commit
- Agent B：用改後的 prompt 生成報告 + 跑 eval
- Agent A 讀 eval 結果 → keep/discard
- 缺點：協調複雜

**方案 C：API Template 模式**
- 不用 Claude Code 生成全文，改用 `POST /api/v1/reports/generate` API
- Agent 只改 TypeScript template（report-generator-local.ts）
- 但這和禁止修改 report-generator-local.ts 衝突

→ 推薦 **方案 A**，但需要注意 context window 管理（每輪 compact）。

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Context window 爆滿 | 每 5 輪手動 compact；報告寫到 /tmp 不佔 context |
| Snapshot 不夠多樣 | 4 個 snapshot 輪流使用，避免 overfit 單一 snapshot |
| Eval 維度太粗 | 5 個 rule-based 指標可能無法捕捉「報告品質」的微妙差異；但至少能防止退化 |
| Agent 自評偏差 | Agent 用自己改的 prompt 生成報告再自己評分——但 evaluator 是 rule-based 不是 LLM，消除了主觀偏差 |
| 和 meeting-prep plan 衝突 | generate-report.md 和 meeting-prep 無交集，不衝突 |

## Review Notes

此計畫比 retrieval 版複雜，因為每輪需要 LLM 生成報告（~3 min）。建議先完成 retrieval autoresearch 的驗證，確認 pattern 有效後再啟動 report 版。

Report eval 目前只有 5 個 rule-based 指標，若要更精確地衡量「報告品質」，未來可擴充 LLM-as-Judge 維度（類似 meeting-prep-eval-l3-l4 的 L3）。
